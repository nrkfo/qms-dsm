# Инструкция по настройке CI/CD (GitHub Self-hosted Runner) на Kubuntu

Данное руководство описывает шаги по установке агента (runner) от GitHub на ваш локальный сервер. Этот агент будет слушать события push в ветку `main` и автоматически запускать локальный скрипт `deploy.sh`. 

Так как сервер находится в защищенной внутренней сети без входящих соединений извне, этот метод идеален — runner сам обращается к GitHub по защищенному каналу и скачивает обновления.

## Шаг 1. Получение токена и команды установки в GitHub
1. Перейдите на страницу вашего репозитория на **GitHub**.
2. Откройте **Settings** -> **Actions** -> **Runners** (в левом меню).
3. Нажмите зеленую кнопку **New self-hosted runner**.
4. Выберите:
   - Runner image: **Linux**
   - Architecture: **x64** (стандартно для Kubuntu).

GitHub сгенерирует для вас уникальные команды. Держите эту вкладку открытой и переходите к терминалу сервера.

## Шаг 2. Загрузка и настройка Runner-а на сервере
Выполните команды в терминале на сервере (можно копировать их со страницы GitHub из Шага 1). Примерный вид команд:

```bash
# Создаем папку для runner'а и переходим в неё (лучше делать это в домашней директории)
mkdir actions-runner && cd actions-runner

# Скачиваем архив с runner'ом (версия может отличаться, скопируйте ссылку из GitHub!)
curl -o actions-runner-linux-x64-2.308.0.tar.gz -L https://github.com/actions/runner/releases/download/v2.308.0/actions-runner-linux-x64-2.308.0.tar.gz

# Распаковываем архив
tar xzf ./actions-runner-linux-x64-2.308.0.tar.gz

# Запускаем конфигурацию
# Важно: Вставьте команду из GitHub (в ней содержится уникальный токен)
./config.sh --url https://github.com/ВАШ_ПОЛЬЗОВАТЕЛЬ/ВАШ_РЕПОЗИТОРИЙ --token ВАШ_СЕКРЕТНЫЙ_ТОКЕН
```

Во время конфигурации вам зададут несколько вопросов:
- **Runner group**: нажмите Enter (оставить Default).
- **Name of runner**: нажмите Enter (оставить имя вашего сервера) или введите `kubuntu-server`.
- **Labels**: нажмите Enter (по умолчанию установится метка `self-hosted`, которая используется в файле `.github/workflows/deploy.yml`).
- **Work folder**: нажмите Enter (оставить `_work`).

## Шаг 3. Установка Runner-а как фоновой службы (Systemd)
Чтобы агент автоматически запускался при старте сервера и постоянно работал в фоне, установите его как сервис:

Находясь в папке `actions-runner`, выполните:

```bash
# Устанавливаем службу (потребуются права sudo)
sudo ./svc.sh install

# Запускаем службу
sudo ./svc.sh start

# Проверяем статус службы
sudo ./svc.sh status
```

Если всё сделано правильно, на странице **Settings -> Actions -> Runners** в GitHub ваш runner должен появиться со статусом **Idle** (зеленый кружок).

## Шаг 4. Предоставление прав на перезапуск служб без пароля sudo (Важно!)
В скрипте `deploy.sh` происходит перезапуск `qms-backend.service` и `nginx` с использованием `sudo`. Чтобы скрипт не завис в ожидании пароля, нужно дать права вашему пользователю.

Выполните команду редактирования конфигурации sudoers:
```bash
sudo visudo
```

Добавьте в самый конец файла следующую строку (если ваш пользователь не `nur`, замените его имя):
```text
nur ALL=(ALL) NOPASSWD: /bin/systemctl restart qms-backend.service, /bin/systemctl reload nginx
```
Сохраните файл и закройте редактор (Ctrl+O, Enter, Ctrl+X для nano).

## Шаг 5. Как это всё теперь работает
1. Разработчик делает коммит и отправляет изменения (`git push origin main`).
2. GitHub видит файл `.github/workflows/deploy.yml` и запускает workflow.
3. Служба агента на вашем сервере Kubuntu перехватывает эту задачу и запускает локальный файл `/home/nur/Рабочий стол/qms-dsm/deploy.sh`.
4. Скрипт `deploy.sh` самостоятельно подтягивает свежий код (`git fetch && git reset --hard`), пересобирает Frontend, обновляет зависимости Backend и мягко перезапускает службы Nginx и Systemd.

> **Важно:** Убедитесь, что вы закоммитили и отправили (push) файлы `.github/workflows/deploy.yml` и `deploy.sh` в ваш репозиторий, чтобы GitHub Actions начал свою работу!
